## 1. 그라운드 룰

> **새로운 대화 세션이 시작될 때마다, 반드시 아래 순서를 따른다.**
> 1. `plan.md`를 읽고 프로젝트의 목적, 구조, 기술 스택, 매니저님 결정 사항을 파악한다.
> 2. `progress.md`를 읽고 현재 진행 단계와 완료/미완료 항목을 확인한다.
> 3. 위 두 문서를 모두 숙지한 후에만 다음 작업 단계를 진행한다.
> 4. 작업 중 발생하는 모든 변경 사항과 매니저님의 결정은 즉시 두 문서에 기록한다.
> 5. **[절대 준수]** 코드든 스크립트든, 채팅창에 보고하기 전에 **무조건 문서에 먼저 기록**한다.
> 6. **[절대 준수]** 모든 설명과 코드 주석은 **한국어**로 작성한다.
> 7. **[절대 준수]** 매니저님이 "개발해", "실행해", "구현해" 등 작업 지시를 내리면, **코딩 전에 반드시 아래 절차를 거친다.**
>    - ① 구현 계획을 **텍스트 + UI 목업(ASCII 또는 스크린샷)** 으로 제시
>    - ② 매니저님의 **명시적 승인("ok", "진행", "ㅇㅇ" 등)** 을 받은 후에만 코딩 시작
>    - ③ 승인 없이 코드를 작성하거나 커밋하지 않는다

## 3. DB 설계

### 테이블 구조
```
members (회원)
├─ id (UUID, PK)
├─ name, phone, email
├─ is_active (BOOLEAN, 기본 TRUE)
└─ created_at

courses (과정)
├─ id (UUID, PK)
├─ name, category(수료증/자격증), description
├─ is_active (BOOLEAN, 기본 TRUE)
└─ created_at

completions (수료 기록)
├─ id (UUID, PK)
├─ member_id (FK → members, CASCADE)
├─ course_id (FK → courses, CASCADE)
├─ issued_date, cohort, note, registration_no
└─ created_at

activity_logs (활동 로그)
├─ id (UUID, PK)
├─ action, target_type, target_id, target_name
├─ details, performed_by
└─ created_at
```

### 특이사항
- 주민번호: DB 제외 (매니저님 결정 2026-03-22)
- 동명이인: 이름 기준 묶기, 추후 연락처로 분리 (매니저님 결정 2026-03-22)
- 가족코칭지도사 2급: 5개 과정 자동 발급 자격증 (별도 수강신청서 없음)

## 6. 병렬 에이전트 운영 체계

### 6-1. 운영 원칙
> 매니저님이 요구사항을 주면, 메인 에이전트가 **작업 분배 보고 → 승인 → 병렬 실행** 순서로 진행한다.
> 각 에이전트는 **담당 파일만 수정**하며, 타 모듈 파일 수정은 금지한다.

### 6-2. 에이전트 구성

| 에이전트 | 역할 | 담당 파일 |
|----------|------|----------|
| **A. 대시보드** | KPI, 차트, 통계 | `app/page.js`, `DashboardClient.js`, `RecentCompletions.js` |
| **B. 회원 관리** | CRUD, 모달, 필터 | `app/members/page.js` |
| **C. 과정+수료** | 과정/수료 현황 | `app/courses/page.js`, `app/completions/page.js` |
| **D. 시스템** | 인증, 로그, 백업 | `app/logs/`, `app/backup/`, `app/login/`, `proxy.js` |
| **E. 테스터** | 읽기 전용 검증 | 수정 금지 — 프리뷰 테스트만 |
| **F. 스킬 매니저** | 스킬 검색/설치 | `find-skills`로 탐색, 에이전트에 활용법 전달 |

### 6-3. 공유 금지 파일 (메인 에이전트만 수정)
`lib/`, `app/components/`, `app/globals.css`, `app/layout.js`, `plan.md`, `progress.md`

### 6-4. 실행 프로세스
1. 매니저님 요구사항 접수
2. **F. 스킬 매니저**가 관련 스킬 검색
3. 메인 에이전트가 **작업 분배표 + 활용 스킬** 보고
4. 매니저님 승인
5. 각 에이전트를 **worktree 격리** 모드로 병렬 실행
6. 메인 에이전트가 **머지 + 충돌 해결 + 문서 반영**
7. push 후 **E. 테스터** 검증
8. **테스터 PASS** → 매니저님에게 완료 보고
9. **테스터 FAIL** → 버그 리포트 생성 → 해당 모듈 에이전트가 수정 → 6번으로 복귀 (최대 3회 반복)

### 6-4-1. 테스터(E) 검증 프로세스
> 테스터는 push 후 프리뷰 서버에서 아래 항목을 검증한다.
> 1. **빌드 성공 여부** — Vercel 배포 에러 없는지
> 2. **콘솔 에러 0건** — 브라우저 콘솔에 에러 없는지
> 3. **기능 동작 확인** — 변경된 기능이 정상 작동하는지
> 4. **기존 기능 깨짐 없음** — 사이드바 네비게이션, 페이지 로딩 등 기본 기능
>
> 검증 결과를 **PASS / FAIL (사유)** 로 보고하며,
> FAIL 시 버그 리포트에 **재현 방법 + 에러 내용**을 포함한다.

### 6-5. 설치된 스킬
- `find-skills` — 스킬 검색
- `nextjs-supabase-auth` — Next.js+Supabase 인증 패턴
- `recharts` — Recharts 차트 패턴

### 6-6. 에이전트 공통 규칙
- 모든 주석 **한국어**
- 담당 파일 외 수정 시 즉시 중단
- `import` 추가 필요 시 메인 에이전트에 요청
- 새 파일 생성은 담당 디렉토리 내에서만
