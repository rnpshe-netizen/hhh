# 프로젝트 설계도 (Plan)

## 🚨 그라운드 룰 (Ground Rule)
> **새로운 대화 세션이 시작될 때마다, 반드시 아래 순서를 따른다.**
> 1. `plan.md`를 처음부터 끝까지 읽고 프로젝트의 목적, 구조, 기술 스택, 매니저님 결정 사항을 파악한다.
> 2. `progress.md`를 읽고 현재 진행 단계와 완료/미완료 항목을 확인한다.
> 3. 위 두 문서를 모두 숙지한 후에만 다음 작업 단계를 진행한다.
> 4. 작업 중 발생하는 모든 변경 사항과 매니저님의 결정은 즉시 `plan.md` 또는 `progress.md`에 기록한다.
> 5. **[절대 준수]** 실제 코드를 짜지 않고 스크립트로 데이터만 조회/검증/단순 확인한 작업조차도, 채팅창에 보고하기 전에 무조건 두 문서(plan/progress)에 먼저 진행 내역을 기록한 뒤 보고한다.
> 6. **[절대 준수]** 개발 과정에서 제시하는 모든 설명(진행 상황, 작업 요약, 도구 사용 이유 등)과 코드 내 주석은 반드시 **한국어**로 작성한다. 매니저님이 기술적 맥락을 완벽히 이해할 수 있도록 쉬운 우리말 사용을 원칙으로 한다.

## 1. 프로젝트 목적
**국제코칭훈련원 회원·과정 통합 관리 시스템** 구축
- 현재 과정별로 흩어져 있는 엑셀 수료 현황을 **하나의 웹 대시보드**에서 통합 관리
- 회원 정보와 수료 이력을 한눈에 파악하고, 축하/안내 메시지를 발송하며, 추후 자격증 자동 발급까지 확장

### 2-3. 목표 상태 (TO-BE)
- **Phase 1** (MVP): 회원 통합 관리 + 과정별 수료 현황 대시보드
- **Phase 2**: 축하 문자 / 과정 추천 안내 (문자, 이메일, 카카오톡)
- **Phase 3**: 수료증·자격증 자동 발급

## 4. Phase 1 (MVP) 세부 계획

### 4-1. 핵심 기능
1. **회원 관리**: 이름, 연락처, 이메일 등 회원 정보 등록/조회/수정/삭제
2. **과정 관리**: 운영 중인 코칭 과정 목록 관리
3. **수료 현황 대시보드**: 회원별 수료 과정 한눈에 보기 (표 + 뱃지 형태)

### 4-3. 웹 페이지 구성
1. **대시보드** (`/`) — 회원별 수료 현황을 뱃지/표로 한눈에 보기
2. **회원 목록** (`/members`) — 회원 검색, 등록, 수정, 삭제
3. **과정 관리** (`/courses`) — 과정 목록 관리

### 7-3. 매칭 전략 (오류 방지 3중 자물쇠)
1. **이름 일치** (필수 — 1차 키)
2. **과정명 일치** (폴더명 ↔ DB 과정명 — 2차 키)
3. **기수 일치** (폴더명 ↔ DB 기수 번호 — 3차 키)

→ 3가지 모두 맞는 경우에만 **자동 매칭**
→ 이름만 맞고 나머지 불일치 → **"수동 확인 필요" 목록에 보류**

### 7-4. 작업 흐름
1. 수강신청서(HWP+DOCX+PDF)에서 이름·전화번호·이메일 일괄 추출
2. 기존 DB 회원과 3중 매칭 실행
3. 매칭 실패 인원 → 출석부 엑셀에서 보조 매칭 시도
4. **검증 리포트 생성** → 매니저님 승인
5. 승인 후 DB 업데이트 (기존값 백업 포함)

### 7-5. 작업 완료 결과 요약 (2026-03-31)
1. **수강신청서(DOCX, HWP, PDF) & 출석부(Excel) 추출**: 총 391개 파일 + 출석부 6종 스캔으로 **573명**의 고유 연락처 획득. HWP 파싱 간 Rust Panic 이슈는 자식 프로세스 독립 실행으로 완벽히 우회 처리.
2. **3중 매칭 (이름+과정+기수)**: 1, 2, 3순위(안전 그룹 + 일부 동명이인 의심 그룹 포함) 도합 **503명** 매칭 성공.
3. **DB 업데이트 완료**: Supabase 서버 기동 상태 확인 후, 503명 전원의 전화번호 및 이메일 정보를 `members` 테이블에 100% 반영 완료 (데이터 검증 성공).

## 9. 병렬 에이전트 운영 체계 (2026-04-01 확정)

### 9-1. 운영 원칙
> 매니저님이 요구사항을 주면, 메인 에이전트(나)가 **작업 분배 보고 → 승인 → 병렬 실행** 순서로 진행한다.
> 각 에이전트는 **담당 파일만 수정**하며, 타 모듈 파일 수정은 금지한다.

### 9-2. 에이전트 구성

| 에이전트 | 역할 | 담당 파일 (수정 가능 범위) |
|----------|------|--------------------------|
| **A. 대시보드** | 대시보드 KPI, 차트, 통계 | `app/page.js`, `app/DashboardClient.js`, `app/RecentCompletions.js` |
| **B. 회원 관리** | 회원 CRUD, 모달, 필터, 검색 | `app/members/page.js` |
| **C. 과정+수료** | 과정 관리, 수료 현황 조회 | `app/courses/page.js`, `app/completions/page.js` |
| **D. 시스템** | 인증, 로그, 백업, 미들웨어 | `app/logs/page.js`, `app/backup/page.js`, `app/login/page.js`, `proxy.js` |
| **E. 테스터** | 읽기 전용 검증, 버그 리포트 | 수정 금지 — 코드 읽기 + 프리뷰 테스트만 |
| **F. 스킬 매니저** | 스킬 검색/설치/활용 전략 | `find-skills`로 필요한 스킬 탐색, 설치, 각 에이전트에 활용법 전달 |

### 9-3. 공유 금지 파일 (동시 수정 불가)
- `lib/supabaseClient.js` — DB 클라이언트 (전 모듈 공유)
- `lib/activityLog.js` — 활동 로그 유틸 (전 모듈 공유)
- `app/components/SidebarNav.js` — 사이드바 메뉴 (메인 에이전트만 수정)
- `app/components/Topbar.js` — 상단바 (메인 에이전트만 수정)
- `app/globals.css` — 전역 스타일 (메인 에이전트만 수정)
- `app/layout.js` — 레이아웃 (메인 에이전트만 수정)
- `plan.md`, `progress.md` — 문서 (메인 에이전트만 수정)

### 9-4. 실행 프로세스
1. 매니저님 요구사항 접수
2. **F. 스킬 매니저**가 요구사항에 맞는 스킬 검색 (`npx skills find`)
3. 메인 에이전트가 **작업 분배표 + 활용 스킬** 작성 후 보고
4. 매니저님 승인
5. 각 에이전트를 **worktree 격리** 모드로 병렬 실행 (스킬 매니저가 추천한 스킬 참고)
6. 완료 후 메인 에이전트가 **결과 머지 + 충돌 해결 + 문서 반영**
7. push 후 **E. 테스터** 에이전트가 검증

### 9-4-1. 스킬 매니저(F) 운영 규칙
- 요구사항 접수 즉시, `npx skills find [키워드]`로 관련 스킬 검색
- **설치 수 1K+ 또는 공식 소스(vercel-labs, anthropics)** 스킬만 추천
- 보안 평가(Gen/Socket/Snyk) 통과한 스킬만 설치
- 설치한 스킬의 SKILL.md를 읽고, 담당 에이전트(A~D)에게 **활용법 요약 전달**
- 이미 설치된 스킬 목록: `find-skills`, `nextjs-supabase-auth`, `recharts`

### 9-5. 에이전트별 공통 규칙
- 모든 코드 주석은 **한국어**로 작성
- 담당 파일 외 수정 시 **즉시 중단** 후 메인 에이전트에 보고
- `import` 추가가 필요한 경우 메인 에이전트에 요청
- 새 파일 생성은 담당 디렉토리 내에서만 가능
